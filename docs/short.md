## 1. 職務要約

クラウド環境の運用が属人化している、コストや権限の実態が把握できていない、監視・バックアップ・障害対応手順が整っていない——こうした状態の会社に入り、現状調査から改善方針の策定・設定変更・手順書化・運用定着まで一人で推進できます。

**直近実績：** GCP環境の運用改善を担当し、月額クラウド費用10%以上削減・不要リソース約20%削除・IAM最小権限化・バックアップおよび障害対応手順の整備を実現。

**経験規模：** 大規模通信インフラ（ソフトバンク）、約200サービス規模のAWS移行支援（BIGLOBE・CCoE）、IoTシェアリングサービスのクラウド基盤（Luup）、社員約200〜300名規模の情シス部門支援・統括。

技術的な対応だけでなく、非エンジニアや経営層への説明・報告にも対応でき、エンジニアリソースが少ない体制でも動ける点が特徴です。

## 2. コアスキル・強み

**【主軸 / 直近実績あり】**

- **GCP運用改善：** Cloud Run / Cloud SQL / Cloud Storage の運用最適化、IAM整理、バックアップ設計、監視・コスト可視化、障害復旧手順整備（直近案件で月額費用10%以上削減・不要リソース20%削除を実現）
- **コスト最適化：** 不要リソース削除、リソースサイズ見直し、月次コスト通知設計、経営層向けコスト削減方針の整理・説明まで対応

**【対応可能 / 実務経験あり】**

- **AWS運用改善：** BIGLOBEでのCCoE活動・約200サービスのAWS移行支援・設計レビュー・ガイドライン整備を経験
- **IAM / セキュリティ整備：** 権限棚卸し・最小権限化・付与削除フロー整備（GCP・AWS・Azure対応）
- **DB負荷対策：** MariaDB / Cloud SQL / RDS を対象に、スロークエリ調査、インデックス整理、接続数・負荷調査、レプリカ / フェイルオーバー方針検討
- **情シス・ガバナンス：** SaaS棚卸し、アカウント管理、ベンダー折衝、少人数情シス部門の運用改善

**【付加価値】**

- **生成AI活用：** Azure OpenAI / Dify / RAG を活用した社内問い合わせボット・FAQ文書検索の設計・導入経験
- **推進力：** 非エンジニアや経営層にも伝わる資料・手順書を作成し、関係部署を巻き込みながら運用定着まで伴走

## 4. フリーランスとしての支援実績

### ■ [業務委託] GCPクラウド運用最適化

| | |
|:---|:---|
| **期間** | 2024年10月〜 |
| **技術スタック** | GCP, Cloud Run, Cloud SQL, Cloud Storage, Cloud Build, Pub/Sub, Workflows, IAM, Cloud Monitoring, BigQuery, Artifact Registry, Secret Manager, Cloud Scheduler, Firebase |
| **役割** | クラウド運用改善、SRE、IAM整理、バックアップ設計、監視・コスト可視化、障害復旧手順整備 |

#### 概要・成果

建設業向けWebサービスを提供する企業にて、GCP環境の運用改善を担当。Cloud Run / Cloud SQL / Cloud Storage を中心に、リソース利用状況の可視化、適正サイズ化、IAM整理、バックアップ・リストア手順、監視・コスト通知、障害復旧手順を整備しました。

> 月額クラウド費用 **10%以上削減** ／ 不要リソース **約20%削除** ／ IAM最小権限化 ／ バックアップ・障害対応手順整備

- 月額クラウド費用を10%以上削減
- 不要リソースを約20%削除し、運用対象と管理コストを整理
- IAM権限の棚卸しと最小権限化により、過剰権限・権限残存のリスクを低減
- バックアップ対象、取得手順、リストア手順を整理し、障害時の復旧可能性を向上
- 障害復旧手順書と復旧訓練により、属人的な障害対応を再現可能な運用へ改善

| | |
|:---|:---|

| **技術スタック** | MariaDB, さくらのクラウド, Webサーバー, PHP, syslog, ログ監視 |
| **役割** | DB負荷調査、スロークエリ調査、運用改善、構成案比較、手順書整備 |

#### 概要・成果

全国にある自動車ディーラーが利用する業務システム / Webアプリケーション基盤において、MariaDBを中心としたDB負荷対策、レスポンス遅延調査、Webサーバー / PHP / DB の同時処理数整理、監視・ログ確認の仕組みづくりを担当しました。

- MariaDB 5台構成におけるレスポンス遅延・接続集中の原因を調査し、ボトルネックを整理
- スロークエリ、インデックス、max_connections、Web / PHP / DB の同時処理数を横断的に確認
- 土日などアクセス増加時に備えた設定見直し方針と運用手順を作成
- レプリカ構成、フェイルオーバー、プロキシ構成の比較資料を作成
- 監視項目、ログ確認手順、障害時の切り戻し手順を整備し、障害対応の属人化を低減
